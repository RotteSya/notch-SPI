# NotchSPI 商业化改造设计 — 题数额度制 · 正式引导 · 设置重构 · 三语化

> 2026-07-10。目标：把 NotchSPI 从"工程师自用工具"变成中国人和日本人都乐意付费使用的
> 成熟商业产品。用户明确授权自主决策、端到端构建，本设计文档为决策留痕。

## 背景与问题

现状（v1.7 + #4/#5 的官方服务）：

1. **引导流程**是一个朴素窗口，三个按钮直接暴露「API Key」「CLI」等技术名词——
   小白用户第一屏就被吓跑。
2. **设置**全部挤在刘海右侧的齿轮弹出菜单里（服务模式/账户/API Key/后端/深度/
   截图目标/快捷键/人物像/更新/退出 全在一个菜单），另有 4 个零散小窗口。
3. **商业模式**是"分钱余额 + 按 Token 计费"（500 分试用、¥/MTok 定价）——
   用户看不懂 Token，对"这次要花多少钱"没有确定感。
4. **无本地化**：全部字符串硬编码中文，日本用户（人物像/性格测试功能的核心受众）
   无法使用。
5. **可自定义项**几乎没有（只有快捷键和讲解深度）。

## 目标

安装 → 走完引导 → 立刻会用，全程不出现 API、CLI、Token、Key 等词。
新用户注册即得 **180 题** 免费额度；每次截屏答题扣 1 题；用完去网页充值题包。
中文/日文/英文三语。每一次交互都精致愉悦。

## 决策一：商业模型 — 题数额度制（替换分钱余额制）

**是什么**：账户余额的单位从"分"改为"题"。1 次成功的截屏问答 = 1 题。
失败（网络错误、模型错误）不扣题——routes 本来就只在成功后扣费，保持。

**为什么赢过按 Token 计费**：定价确定性。用户按下快捷键之前就知道代价是"1 题"，
和"180 题免费"形成同一套心智。Token 计费的不确定感是充值转化的最大心理障碍。

**服务端**（server/，契约 v2，产品未上线无兼容包袱，干净切换）：

- `devices` 表：`balance_cents` → `balance_questions`（整数题数），新增
  `total_questions` 累计已答题数。Token 累计列保留（内部成本核算用）。
- 注册赠送 `TRIAL_QUESTIONS = 180`。
- `POST /v1/devices` → `{ device_token, balance_questions: 180 }`
- `GET /v1/account` → `{ balance_questions, total_questions, total_input_tokens, total_output_tokens }`
- `POST /v1/captures`：余额 ≤ 0 题 → 402；成功后扣 1 题；SSE usage 事件：
  `{ "type":"usage", "input_tokens":…, "output_tokens":…, "questions_charged":1, "balance_questions":179 }`
- 充值页 `/topup` 改为**题包**购买（题包目录在服务端 config：如 100题/300题/1000题，
  金额与币种由运营配置；支付桩保留 stub 流程，生产换 Stripe/支付宝/微信）。
- `usage_events` 记录 tokens + questions；`topups` 记录 `amount_questions` + 实付金额。

**客户端**（Cloud/）：

- `OfficialAPI.balanceCents` → `balanceQuestions`；`formatBalance` → "180 题" 格式。
- `BillingGate` 按剩余题数拦截，文案零术语（"额度已用完，去充值后继续"）。
- 完成状态行："完成 · 剩余 179 题"；剩余 ≤ 10 题时状态行温和提醒。
- 旧的 `official.balanceCents` 缓存键直接废弃（未上线，无迁移负担）。

## 决策二：正式引导流程（Onboarding v2）

替换现有单屏三按钮窗口。新形态：**无边框圆角暗色窗口（560×420）**，与刘海同一套
黑曜石设计语言，背景是一层**自绘极光流动着色器**（CAMetalLayer + 自定义 fragment
shader；机器不支持 Metal 时退化为静态渐变）。步骤页横向滑动过渡 + 进度点：

1. **欢迎** — Rose 曲线动画 + 品牌名 + 一句话价值主张；右上角语言切换
   （自动检测系统语言预选：zh-Hans / ja / en）。
2. **它如何工作** — 三格图解：按快捷键 → 屏幕被读取 → 答案从刘海流出。
   动画演示刘海展开的微缩模型。
3. **屏幕录制权限** — 解释为什么需要（"为了看到你屏幕上的题目"），
   一键打开系统设置；`CGPreflightScreenCaptureAccess` 实时轮询，授权后自动亮绿打钩。
4. **礼物时刻** — 后台静默注册设备（官方服务默认，不问用户），成功后
   数字滚动动画展示"180 题已到账" + 粒子庆祝动效。注册失败不阻塞：
   显示"额度将在首次使用时自动到账"，首次截屏时会自动重试注册。
5. **立即试用** — 大字显示当前快捷键，提示"现在就按下试试"；检测到第一次
   capture 或点击"开始使用"即完成。

全程无 API/CLI/Key/Token 字样。自定义 Key 和本机 CLI 两个高级模式移入
设置 → 高级，供极客自己发现。

## 决策三：统一设置窗口

一个真正的 macOS 设置窗口（侧栏导航 + 内容区，650×480），六页：

| 页 | 内容 |
|---|---|
| 通用 | 语言（跟随系统/中/日/英）、讲解深度默认值、截图目标 |
| 快捷键 | 现有三条录制行（迁移） |
| 外观 | 强调色（长春花/樱/抹茶/琥珀/月白 5 选）、答案字号（小/标准/大）、自动收起时长、答题后展示时长 |
| 账户与额度 | 剩余题数环形图（动画渐变描边）、累计已答、充值按钮、设备 ID、重试初始化 |
| 人物像 | 现有 PersonaManager（迁移） |
| 高级 | 服务模式三选（官方/自定义 Key/本机 CLI）、API Key 编辑（迁移）、检查更新、版本号 |

齿轮菜单精简为高频快速项：讲解深度、截图目标、切换人物像、账户余额一眼可见、
「设置…」、退出。所有原独立小窗（快捷键/人物像/API Key/账户）并入设置窗对应页。

## 决策四：三语本地化（zh-Hans / ja / en）

自建轻量 `L10n` 层（enum + 三语字典，类型安全），不用 .lproj：

- 运行时即时切换（.lproj 需要重启且跟系统语言走，不满足"引导页第一屏选语言"）。
- 字符串总量 ~200 条，代码内字典完全可控、可 grep、无资源管线。
- `L10n.current`：`auto`（跟随系统首选语言映射到三语之一，默认 en 兜底）或手动指定；
  切换发 Notification，所有窗口刷新。
- 服务端错误消息保持中文源串 → 客户端对**已知错误类别**用错误码本地化，
  未知错误原样透传（v2 契约给 error 加 `code` 字段：`insufficient_quota` /
  `invalid_token` / `upstream_error`）。

## 决策五：外观自定义与打磨

- `NotchPalette.accent` 动态化：读设置，5 款策展色。Rose、胶囊、边缘高光、
  环形图全部跟随。
- 答案字号三档接入 `NotchType`。
- 自动收起延时（3s/9s/常驻）接入 NotchController。
- 额度 ≤10 题：完成状态行附加提示；账户页环形图变琥珀色。

## 错误处理原则

- 引导每一步都可跳过/失败不阻塞（权限可以稍后给、注册可以稍后重试）。
- 官方服务任何失败都有一条不带术语的出路文案 + 自动重试点（首次截屏）。
- 高级模式的报错保持技术细节（受众是极客）。

## 测试与验证

- `PureLogicTests` 扩展：BillingGate 题数逻辑、usage 事件解析（questions 字段）、
  L10n 完整性（三语 key 对齐）、题数格式化。
- server 全部测试改题数断言；新增题包/402/усage 事件测试。
- 视觉验证：DEBUG `--visual-qa` 启动，重置 defaults 走完整引导截图每一步；
  设置六页截图；三语各截引导首屏；刘海收起/展开/流式/低额度四态截图。

## 非目标（本次不做）

- 真实支付网关接入（保留 PaymentProvider 接缝与 stub）。
- 账户跨设备同步/登录体系（匿名设备令牌足够 v1 商业化）。
- App Store 分发（继续 DMG + 公证通道）。
